# Architecture Sprint 1. Часть 1

## Наблюдения по коду

### Компоненты

#### App
В основном компоненте `App` есть общие `Header` и `Footer`:
* `Footer` всегда одинаковый
* `Header` зависит от текущего пути, но в нем нет никаких зависимостей от других компонент

Основная часть странички - либо компонент `Main`, либо `Login`/`Register`, рутинг зависит от наличия (и валидности) jwt-токена.

Также важной частью `App` являются поп-апы, которые можно сразу сгруппировать:
* Профиль пользователя
    * `EditProfilePopup` - редактирование профиля пользователя (имя и занятие)
    * `EditAvatarPopup` - редактирование аватара пользователя
* Места (`places`)
    * `AddPlacePopup` - добавление новой карточки места
    * `PopupWithForm` - подтверждение удаления карточки
    * `ImagePopup` - поп-ап с увеличенной карточкой (открывается после нажатия на карточку места)
* Аутентификация
    * `InfoTooltip` - поп-ап с уведомлением при регистрации или входе

#### Main
В `Main` явно выделяются две секции - профиль и галерея (`places`)

#### Popups
Важное наблюдение: поп-апы хоть и легко разделяются по бизнес-логике на группы, но содержат общие зависимости - компонент `PopupWithForm`.

### Стили
Стили в проекте тоже хорошо разделены на блоки, за исключением двух общих:
* `blocks/page` - `page__section` будет использоваться в профиле и в `places`
* `blocks/popup` - общие стили для всех поп-апов

### API
Есть всего 2 файла для взаимодействия с бэкендом, посмотрим на них по отдельности

#### `utils/auth.js`
В этом файле 3 запроса: `login` и `register`, которые будут использоваться в компонентах `Login` и `Register`, соответственно, а также `checkToken`, вызываемый один раз при монтировании компонента `App` для проверки jwt-токена из `localStorage`'а.

#### `utils/api.js`
Снова методы (кроме одного `getAppInfo`) легко группируются по бизнес-логике, к которой они относятся:
* Профиль пользователя
    * `getUserInfo` - получение информации о пользователе
    * `setUserInfo` - обновление информации о пользователе
    * `setUserAvatar` - загрузка аватара
* Места
    * `getCardList` - получение списка карточек мест
    * `addCard` - добавление новой карточки места
    * `removeCard` - удаление карточки
    * `changeLikeCardStatus` - удаление и постановка лайка на карточку

## Выводы из наблюдений
Можно сходу выделить 3 отдельных модуля, которые станут независимыми микрофронтендами:
* `profile` - профиль пользователя, включающий в себя секцию профиля из `Main.js`, соответствующие стили, поп-апы и API-методы из `api.js`
* `places` - управление "местами", включает компонент для отображения карточек и лайков, соответствующие поп-апы и методы из `api.js` для взаимодействия с карточками и управления лайками
* `auth` - аутентификация пользователя, компоненты для входа и регистрации, а также файл `utils/auth.js` для взаимодействия с бэкендом

Отдельно хочется упомянуть общий код, к которому относятся стили из `blocks/page` и `blocks/popup`, а также базовый компонент для большинства поп-апов - `PopupWithForm`. В этом месте однозначно хочется избежать дублирования кода по двум микрофронтендам (`profile` и `places`), поэтому очень хотелось вынести это в отдельную либу (`microfrontend/libs`). Однако, хоть для стилей/картинок это и получилось, вынести в отдельную библиотеку React-компонент `PopupWithForm` у меня не вышло... Потратив на это достаточно времени, решил все же продублировать этот компонент во всех микрофронтендах, но это точно можно правильно настроить (я надеюсь).

## Разделение на микрофронтенды
Для начала необходимо определиться с фреймворком

### Выбор фреймворка
Во-первых, основная особенность `Single SPA`, связанная с независимостью от фреймворков, в этом случае нам ничего не дает, т.к. разделяемое приложение написано целиком на реакте.

Во-вторых, в `Single SPA` интеграция микрофронтендов на хосте выглядит как routing в зависимости от текущего пути. В случае фронтенда `Mesto` это выглядит неудобно, потому что надо совмещать на одной странице компоненты `profile` и `places`, что может вызвать дополнительные трудности в реализации. Фреймворк `Module Federation` для такого кейса подходит лучше.

Ну и в целом, переход на `Module Federation`, как сейчас кажется, потребует меньше усилий в изменении существующего кода.

**Остановимся на `Webpack Module Federation`**.

### Запуск микрофронтендов

Сначала необходимо запустить все 3 зависимости хоста:

1. Запускаем `auth-microfrontend`
   ```
   cd microfrontend/auth-microfrontend
   npm i
   npm start
   ```
2. Запускаем `places-microfrontend`
   ```
   cd microfrontend/places-microfrontend
   npm i
   npm start
   ```
3. Запускаем `profile-microfrontend`
   ```
   cd microfrontend/profile-microfrontend
   npm i
   npm start
   ```

И, наконец, теперь можно запускать хост. Он находится в корневой директории проекта:
```
npm i
npm start
```

После этого на `localhost:8080` должен быть полностью рабочий сайт.
